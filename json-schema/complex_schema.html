构建复杂的Schema

通常在构建写代码是，我们通常会考虑“结构化”程序代码， 通过可重用的方法块来避免简单的复制黏贴造成的问题。

同样我们也可以在JSON Schema中也可以将有用的结构分成各个可重用的部件。

重用

在这个例子中，我们假设有一个客户记录(customer)，该记录中的客户可能同时包含邮寄地址与账单地址。 地址的结构总是相同的，它们有street address, city和state， 因此我们不想在每个需要存储地址的地方重复定义这些结构。 不仅因为这样做会使得整个Schema变得臃肿，而且使得在将来进行结构更新时变得非常麻烦。 如果我们以后需要在地址结构中增加country(国家)字段，那么我们必须修改所有重复定义的地址结构。

所以我们需要在一个地方去定义一个可重用的地址结构，然后在用到的地方进行重用。

首先我们定义一个地址标准结构。

{
  "type": "object",
  "properties": {
    "street_address": { "type": "string" },
    "city":           { "type": "string" },
    "state":          { "type": "string" }
  },
  "required": ["street_address", "city", "state"]
}

既然我们打算重用这样的schema，我们通常（非必须）将它放置在父级schema的definitions定义中：

{
  "definitions": {
    "address": {
      "type": "object",
      "properties": {
        "street_address": { "type": "string" },
        "city":           { "type": "string" },
        "state":          { "type": "string" }
      },
      "required": ["street_address", "city", "state"]
    }
  }
}

我们可以从任何地方通过$ref关键字来引用到这个schema。最简单的方法是将$ref的值设定为该address的逻辑地址。 如下：

{ "$ref": "#/definitions/address" }

$ref的值是一个字符串，称为JSON Pointer

JSON Pointer与XML中的XPath具有相似的概念，但逻辑更为简单

井号#代表当前的文档，斜杠/分隔了到达目标文档的移动路径。因此在我们的例子中 #/definitions/address代表着：

  1. 移动到文档的根部
  2. 找到键值definitions
  3. 在definitions这个对象中，找到address

$ref的值可以是相对路径，也可以是绝对路径。 所以如果你将你的结构定义在单独的文件中，你可以通过如下的方式进行引用。

{ "$ref": "separate_file.json#/address" }

这样将会在当前文件的旁边装载address schema。

现在我们可以把所有东西结合在一起定义我们的customer schema

{
  "$schema": "http://json-schema.org/draft-04/schema#",

  "definitions": {
    "address": {
      "type": "object",
      "properties": {
        "street_address": { "type": "string" },
        "city":           { "type": "string" },
        "state":          { "type": "string" }
      },
      "required": ["street_address", "city", "state"]
    }
  },

  "type": "object",

  "properties": {
    "billing_address": { "$ref": "#/definitions/address" },
    "shipping_address": { "$ref": "#/definitions/address" }
  }
}

如下的json数据可以验证通过

{
  "shipping_address": {
    "street_address": "1600 Pennsylvania Avenue NW",
    "city": "Washington",
    "state": "DC"
  },
  "billing_address": {
    "street_address": "1st Street SE",
    "city": "Washington",
    "state": "DC"
  }
}

扩展性

当$ref与$allOf、$anyOf和$oneOf结合时，凸显出其价值。

我们以shipping address（邮寄地址）为例，我们想要知道改地址是一个住宅地址(residential）还是一个办公地址(business)， 因为不同类型的地址配送方式不同。而对于billing address（账单地址）我们不需要标注地址类型。

为了解决这个问题，我们需要调整shipping address的定义。

"shipping_address": { "$ref": "#/definitions/address" }

可以通过使用allOf关键字组合核心的address schema定义与额外的地址类型描述。如下：

"shipping_address": {
  "allOf": [
    // Here, we include our "core" address schema...
    { "$ref": "#/definitions/address" },

    // ...and then extend it with stuff specific to a shipping
    // address
    { "properties": {
        "type": { "enum": [ "residential", "business" ] }
      },
      "required": ["type"]
    }
  ]
}

完整的schema如下：

{
  "$schema": "http://json-schema.org/draft-04/schema#",

  "definitions": {
    "address": {
      "type": "object",
      "properties": {
        "street_address": { "type": "string" },
        "city":           { "type": "string" },
        "state":          { "type": "string" }
      },
      "required": ["street_address", "city", "state"]
    }
  },

  "type": "object",

  "properties": {
    "billing_address": { "$ref": "#/definitions/address" },
    "shipping_address": {
      "allOf": [
        { "$ref": "#/definitions/address" },
        { "properties":
          { "type": { "enum": [ "residential", "business" ] } },
          "required": ["type"]
        }
      ]
    }
  }
}

如下的json值不合规，因为缺少了地址类型的描述：

{
  "shipping_address": {
    "street_address": "1600 Pennsylvania Avenue NW",
    "city": "Washington",
    "state": "DC"
  }
}

合规的json值如下：

{
  "shipping_address": {
    "street_address": "1600 Pennsylvania Avenue NW",
    "city": "Washington",
    "state": "DC",
    "type": "business"
  }
}

通过基础结构的组合，我们可以很方便的构建出复杂有效的结构来，同时又避免了重复。
